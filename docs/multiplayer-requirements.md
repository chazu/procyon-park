## 0. Document Scope & Conventions

This document captures **requirements** — WHAT the system must do and WHY — for
turning Procyon Park (PP) from a single-user orchestrator into a multi-user
collaborative harness with distributed agent execution. It does **not** specify
schemas, wire formats, or step-by-step implementation.

Requirements use RFC-2119 keywords (**MUST**, **SHOULD**, **MAY**, **MUST NOT**)
where priority matters. Every feature includes acceptance criteria phrased as
verifiable behaviors against named code surfaces:

- BBS API: `src/bbs/BBS.mag`, categories enumerated in `src/bbs/Categories.mag`
- Tuple factory: `src/bbs/Tuple.mag`
- HTTP routes: `src/api/Server.mag`
- SSE: `src/api/DashboardSSE.mag`
- Dispatcher/scheduler: `src/dispatcher/Dispatcher.mag`, `Scheduler.mag`
- Harness: `src/harness/ClaudeHarness.mag`, `HarnessFactory.mag`
- CLI: `src/cli/PP.mag`, `src/cli/CliPP.mag`
- Roles: `src/roles/*.mag`
- Workflows: `workflows/*.cue`
- Maggie Ed25519 primitives: `<maggie-repo>/vm/dist/identity.go`

## 1. Decided Architecture (Inputs)

The following are **settled** and out of scope for debate in this document:

1. **Centralized PP server.** One canonical PP process hosts the BBS tuplespace
   (`src/bbs/BBS.mag`), the dispatcher tick loop (`src/dispatcher/Dispatcher.mag`),
   the workflow engine, and the HTTP+SSE API. Rationale: the goal is **collective
   improvement of the harness itself**, which requires a single authoritative
   pool of observations, facts, conventions, and decisions — not N federated
   pools that drift.

2. **Pull-based worker model.** The central server does **not** spawn Claude
   processes. The existing scheduler-driven local harness spawn path
   (`Scheduler>>dispatchTask:` → `HarnessFactory>>createFor:` → `ClaudeHarness>>run`)
   is retained for `pp serve --local` but is supplemented by a new path in
   which remote `pp worker` daemons long-poll the task queue, claim tasks,
   spawn Claude locally with the user's own Anthropic API key, and stream
   results back as BBS tuples over HTTP.

3. **Ed25519 per-user identity.** Every human has a keypair generated by
   `pp identity init`, stored under `~/.config/pp/identity`. The server
   registers public keys via `pp user add <name> --pubkey <hex>` into a new
   pinned BBS category `identity` (added to `Categories>>valid` and
   `Categories>>pinned`). The CLI signs every mutating HTTP request; the server
   verifies and stamps an `actor` field on every mutation. The signing layer
   reuses `maggie/vm/dist/identity.go` (`NodeIdentity.Sign` / `Verify`).

4. **Derived agent identities.** Agents run under scoped actor strings of the
   form `<human>/<workflow-instance>/<role>` (e.g.
   `alice/scout-mission-1776559250/scout`). The worker daemon (not Claude)
   holds the signing key; agent tuples trace back to both the launching human
   and the executing worker.

5. **Delegated execution is a property of pull-based workers.** Task tuples
   gain `launched-by` (human actor) and `executed-by` (worker id). Bob's worker
   MAY execute Alice's scout-mission subject to affinity and policy rules.

## 2. Goals

### 2.1 Collective improvement of the harness

The purpose of multi-user PP is to make the harness itself get better faster
by pooling human and agent effort. Concretely, "collective improvement" means:

- **G1.1** All users share a single observation feed. Observations recorded by
  Alice's agents (category `observation`) MUST be visible to Bob through the
  same `/api/scan` and SSE endpoints, subject only to scope filtering.
- **G1.2** Facts (`category: fact`), conventions (`category: convention`),
  decisions (`category: decision`), and rules (`category: rule`) are pinned
  on a shared tuplespace and visible to every registered user's agents.
- **G1.3** Every mutation to pinned knowledge categories MUST be attributed to
  a human actor via the `actor` field, so contribution to the harness is
  traceable.
- **G1.4** Conventions promoted from repeated observations (the
  observation → fact → convention pipeline referenced in `src/bbs/Categories.mag`)
  MUST surface in every user's agent context, not just the observer's.

### 2.2 Team situational awareness

- **G2.1** A user looking at the dashboard (`/dashboard/`) MUST be able to see
  which other humans are online, which workers are online, what workflows are
  running, who launched them, and which worker is executing each task.
- **G2.2** `pp notify` (see `handleNotify` in `Server.mag`) MUST fan out to
  every subscribed observer, not just the invoking user.

### 2.3 Distributed compute with cost attribution

- **G3.1** Agent execution MUST be able to occur on any worker-enabled machine
  in the team, not only the PP server host.
- **G3.2** Token/API usage MUST be attributable to the worker that spent it
  (and therefore to the Anthropic API key owner) independent of who launched
  the task.
- **G3.3** Workers with heterogeneous capabilities (different models
  configured, different repos cloned, different slot counts) MUST coexist.

### 2.4 Delegated execution

- **G4.1** A human MUST be able to launch a workflow whose tasks execute on
  another human's worker, provided both sides' policies permit it.
- **G4.2** Every delegated task's provenance MUST be recorded: `launched-by`
  AND `executed-by` fields are both present on the `task` tuple.

### 2.5 Secure multi-tenant use

- **G5.1** Every mutating endpoint on `src/api/Server.mag` MUST reject requests
  that are unsigned, expired (timestamp older than a skew window), or signed
  by a public key not registered in the `identity` category.
- **G5.2** The `actor` stamped by the server MUST be derived from the verified
  signature, never from a client-supplied field.

---

## 3. Non-Goals (Explicit)

- **N1** High availability / zero-downtime server. v1 assumes a single PP
  server process. If it is down, all work stops. No clustering, no leader
  election, no failover. Operators MAY restart the server; the BBS JSONL
  persistence layer provides crash recovery.
- **N2** Distributed tuplespace / true Linda federation. Gated on the Maggie
  distributed-tuplespace work (see `multiplayer-research.md §1` — absent
  today). The tuplespace remains a single-process `TupleSpace` inside the
  central PP.
- **N3** Anonymity. All mutating actions are attributed to a registered
  identity. Read-only public browsing MAY exist later but is not v1.
- **N4** Shared filesystem / server-side repo mirror. Workers self-declare the
  repos they hold locally; the server does not fetch, mirror, or synchronize
  source code.
- **N5** Cross-worker message passing. Workers talk only to the central server.
- **N6** Public-internet exposure hardening. v1 assumes a same-team trust
  boundary (LAN, VPN, or Tailscale). Threat model does not include anonymous
  attackers with line-of-sight to the port.
- **N7** Web-of-trust / PKI infrastructure. Public keys are registered by an
  operator running `pp user add`. No CA, no signed key endorsements.

## 4. Features

### 4.1 Identity Management

**F1.1 Keypair generation — `pp identity init`**
- MUST generate an Ed25519 keypair using `maggie/vm/dist/identity.go`
  (`NewNodeIdentity`) semantics.
- MUST write the private key with mode `0600` to
  `~/.config/pp/identity/<name>.key` and the public key as hex to
  `~/.config/pp/identity/<name>.pub`.
- MUST refuse to overwrite an existing key without `--force`.
- SHOULD print the public key in both hex and proquint form (the proquint
  helper exists at `NodeIdentity.NodeIDProquint`).

*Acceptance:* Running `pp identity init alice` twice without `--force` exits
non-zero and leaves the original key intact.

**F1.2 User registration — `pp user add <name> --pubkey <hex>`**
- MUST write a pinned tuple in category `identity`, scope `'/'`, identity
  `user:<name>`, payload `{ pubkey, created_at, status: 'active' }`.
- The category `identity` MUST be added to both `Categories>>valid` and
  `Categories>>pinned` in `src/bbs/Categories.mag`.
- MUST reject duplicate names unless `--replace` is passed (key rotation
  path).

*Acceptance:* After `pp user add alice --pubkey ...`, a `GET /api/scan` for
category `identity` returns exactly one tuple whose payload contains Alice's
pubkey.

**F1.3 Request signing — CLI side**
- The CLI (`src/cli/PP.mag`, `CliPP.mag`) MUST, for every mutating HTTP call
  (`/api/out`, `/api/inp`, `/api/observe`, `/api/notify`,
  `/api/workflow/*`, `/api/workitem/*`), append headers:
  - `X-PP-Actor: <name>`
  - `X-PP-Timestamp: <unix-seconds>`
  - `X-PP-Signature: <hex(ed25519_sign(private_key, canonical_bytes))>`
- The canonical byte string signed MUST include method, path, timestamp,
  and body hash. Specifics are a design concern; the **requirement** is
  that replay across endpoints or bodies fails verification.

**F1.4 Request verification — server side**
- The server MUST reject any mutating request where signature verification
  fails, the timestamp skew exceeds a configurable window (default 120 s),
  or the actor is not present/active in the `identity` category.
- On success, the server MUST stamp `actor = <verified name>` on every
  tuple it writes on behalf of that request (passed through `BBS>>out:`
  and friends; or added in `Server.mag` handler wrappers).
- GET endpoints MAY be unauthenticated for v1 (dashboards/observability),
  but MUST NOT accept any state-changing verb.

*Acceptance:* A replayed request with an old `X-PP-Timestamp` returns 401;
a request signed by an unregistered key returns 401; a valid request's
resulting tuple has `actor` matching the signing identity.

**F1.5 Key rotation — `pp user add <name> --pubkey <new> --replace`**
- MUST update the existing `identity` tuple's payload to the new pubkey and
  bump `rotated_at`.
- The old key MUST stop verifying immediately. (See Open Question §6.9.)

**F1.6 Key revocation — `pp user revoke <name>`**
- MUST set payload `status: 'revoked'` on the user's identity tuple.
- Revoked identities' signatures MUST be rejected by F1.4. Existing tuples
  authored by a revoked identity are retained (audit trail).

### 4.2 Worker Daemon — `pp worker`

**F2.1 Registration**
- On startup, a worker MUST POST a signed registration to
  `/api/worker/register` including its worker id (derived from a separate
  worker keypair or from the operator's human keypair — see §6.x), a
  capability manifest (see F2.2), and a heartbeat interval.
- The server MUST record the worker as a tuple in a new category
  `worker` (affine with TTL = 3× heartbeat interval), scope `'/'`,
  identity `worker:<id>`.

**F2.2 Capability declaration**
- The manifest MUST include:
  - `slots` (int, concurrent harness slots)
  - `models` (array of strings, e.g. `['claude-opus-4.7', 'claude-sonnet-4.7']`)
  - `repos` (array of `{name, path, head_commit}` — self-declared clones)
  - `roles` (array of role names the worker is willing to run; defaults to
    all in `src/roles/`)
  - `operator` (human actor that vouches for the worker)
  - `accept_from` (array of human actors whose tasks this worker will claim;
    see F2.7)
- The manifest MUST be refreshable: a worker that clones a new repo can
  re-register without restarting in-flight tasks.

**F2.3 Heartbeat**
- The worker MUST POST `/api/worker/heartbeat` at least once per heartbeat
  interval.
- The server MUST mark workers whose affine tuple expired as offline;
  offline workers MUST NOT be assignable new tasks.

**F2.4 Graceful drain — `pp worker drain`**
- The worker MUST stop claiming new tasks, finish in-flight tasks, then
  exit with status 0.
- The server MUST surface drain state on the dashboard.

**F2.5 Budget guardrails**
- A worker MUST support `--max-tokens-per-day N`, `--max-usd-per-day N`, or
  both.
- On exceeding any configured budget, the worker MUST enter a soft-drain
  state: stop claiming tasks, continue in-flight, and emit a `notification`
  tuple with severity `warning`.
- See Open Question §6.8 for hard/soft semantics.

**F2.6 Task claiming flow**
- The worker MUST long-poll `/api/tasks/available?worker=<id>` (an SSE or
  long-poll endpoint added to `src/api/Server.mag`) with its capability
  manifest digest so the server can filter.
- Claiming a task MUST be atomic on the server side, implemented as a
  `BBS>>update:` transaction on the `task` tuple that flips
  `status: 'pending' → 'claimed'` and sets `executed-by: worker:<id>` and
  `claim_expires_at`. This extends the existing pattern in
  `Scheduler>>dispatchTask:`.
- If two workers race, exactly one MUST succeed and the loser MUST receive
  a "claim conflict" response.

**F2.7 Accept-from allowlist**
- A worker's `accept_from` field defaults to `['*']` (any registered user).
- If set, the worker MUST refuse to claim tasks whose `launched-by` is not in
  the list. Enforcement happens server-side during claim matching (workers
  cannot be trusted to self-police against a compromised operator's policy,
  so the server consults the worker's registered manifest, not the worker's
  claim request).

### 4.3 Task Claiming & Lease

**F3.1 Eligibility matching**
- The server MUST only offer a task to a worker when all of the following
  hold:
  - The task's `affinity` (§4.4) is compatible with the worker's manifest.
  - The task's required role is in the worker's `roles`.
  - The task's required model is in the worker's `models`.
  - The task's repo (if any) is in the worker's `repos` at the required
    commit (or any commit, per Open Question §6.5).
  - The worker is not offline, not draining, not over budget.
  - The task's `launched-by` passes the worker's `accept_from` policy.

**F3.2 Claim lease**
- A successful claim MUST carry a `claim_expires_at` timestamp, default 5
  minutes from claim time. The exact default is an Open Question (§6.2).
- The worker MUST renew the lease while the task is in-flight via
  `/api/tasks/renew` at some cadence shorter than the expiry.

**F3.3 Claim expiry & reaping**
- A reaper loop (part of `Dispatcher>>runLoop`) MUST sweep tasks whose
  `claim_expires_at` has passed and `status = 'claimed' | 'running'` but
  no completion tuple has arrived, revert them to `pending`, increment a
  `retry_count`, and emit an `event` tuple describing the reap.
- After a configurable `max_retries` (default 3), a task MUST transition to
  `status: 'failed'` with payload noting reaping as the cause.

**F3.4 Conflict resolution**
- If a worker reports completion for a task that has already been reaped and
  re-claimed by another worker, the server MUST record the late completion
  as an `event` tuple (`category: event`, identity `late-completion:<task>`)
  but MUST NOT overwrite authoritative task state.

### 4.4 Task Affinity & Routing

**F4.1 Affinity tuple field**
- The `task` payload MUST support an `affinity` dictionary with optional
  keys:
  - `launcher_only: true` — only workers whose `operator == launched-by`
    may claim.
  - `workers: ['worker:<id>', …]` — explicit allowlist.
  - `model: 'opus' | 'sonnet' | <concrete>` — routes by model.
  - `repo: '<repo-name>@<commit>'` — routes by local repo availability.
  - `team: true` — any active worker (the default).

**F4.2 Workflow-level affinity**
- Workflows (CUE files in `workflows/`) MUST be able to set default affinity
  for their tasks. This is an addition to the CUE schema, not a new
  runtime feature per se, but is listed here because it is the primary
  way affinity gets onto task tuples.

**F4.3 CLI affinity override**
- `pp run <workflow>` MUST accept `--affinity launcher-only`,
  `--affinity team`, `--worker <id>`, `--model <name>`, `--repo <name>`.

### 4.5 Attribution

**F5.1 Tuple provenance fields**
- The `Tuple` factory (`src/bbs/Tuple.mag`) MUST, for all tuples written
  through signed endpoints, carry:
  - `actor` — the verified human actor (already defined).
  - `launched-by` — on tasks, the actor that initiated the workflow.
  - `executed-by` — on tasks, the worker id that ran the task (nil until
    claimed).
  - `witnessed-by` — on observations, the actor(s) for whom this observation
    was made (see Open Question §6.1).
- These fields MUST be server-stamped from verified state, never from
  client-supplied bodies (except `witnessed-by`, which is client-supplied
  but additionally constrained).

**F5.2 Agent-identity derivation**
- The harness (whether local or on a remote worker) MUST set its BBS
  `identity` string to `<human>/<workflow-instance>/<role>` — e.g.
  `alice/scout-mission-1776559250/scout`. This is the value already
  threaded through `Identity>>taskComplete:task:`, etc., in
  `src/bbs/Identity.mag`.
- Signatures on agent-authored tuples MUST be made by the *worker's* key,
  not by Claude. Agents themselves never see private key material.

### 4.6 Audit Trail & Provenance Queries

**F6.1 Contribution query**
- `pp audit --actor alice` MUST return a summary: tuples written by
  actor `alice`, broken down by category and by workflow.
- Implementation leverages `BBS>>scanAll:` and a new `--actor` filter on
  `/api/scan`.

**F6.2 Workload query**
- `pp audit --worker worker:<id>` MUST return tasks executed by the given
  worker, with token/time totals if recorded.

**F6.3 Workflow lineage**
- `pp audit --workflow <instance-id>` MUST return the full chain: launching
  human, each task, each executing worker, each child workflow.

### 4.7 Dashboard Multi-User Presence

**F7.1 Online humans**
- The dashboard (served from `static/`, data via
  `src/api/DashboardSSE.mag`) MUST render a presence panel listing humans
  currently connected to the SSE stream.

**F7.2 Online workers**
- The presence panel MUST list active `worker` tuples with their capability
  manifest digest (slots in use / total, models, drain state, budget state).

**F7.3 Per-card attribution**
- Every workflow card (currently shows role badges, per recent commits on
  `main`) MUST additionally show the launcher and, per task, the executing
  worker.

**F7.4 Scope filter**
- The dashboard MUST offer a view toggle: "all users" vs "mine only".
  Default is an Open Question (§6.7).

### 4.8 Multi-Observer `pp notify` Fan-Out

**F8.1** The `handleNotify` endpoint in `Server.mag` currently emits a
`notification` tuple and surfaces it on the local SSE stream. Under
multi-user, notifications MUST be delivered to every SSE subscriber whose
identity is either:
  - the `launched-by` of the workflow the notification references, **or**
  - explicitly listed in a new `observers` field on the notification, **or**
  - subscribed to the workflow instance via `pp watch <instance-id>`.

**F8.2** A human MAY `pp watch <instance-id>` / `pp unwatch <instance-id>` to
opt in/out of another user's workflow notifications.

### 4.9 Local-Mode Fallback — `pp serve --local` (v2)

**F9.1** A non-goal for v1 in terms of new behavior, but the existing
scheduler path (local harness spawn via `HarnessFactory>>createFor:`) MUST
remain functional and MUST be reachable via `pp serve --local`, which runs
the server with identity/signature verification disabled and spawns
harnesses in-process exactly as today. This is the regression-safety
escape hatch for single-user users.

## 5. Non-Features (Explicit)

- **NF1** No server-side Claude execution. The PP server process MUST NOT
  hold an Anthropic API key, MUST NOT spawn Claude CLI processes (outside
  `--local` mode), and MUST NOT proxy Claude API calls.
- **NF2** No cross-worker direct communication. Workers communicate only
  with the central server. There is no NAT traversal story, no peer
  channel.
- **NF3** No shared filesystem or code mirror. Workers declare what they
  have; the server does not fetch, clone, diff, or host source.
- **NF4** No OAuth/SSO/password login. Only Ed25519 keypairs registered via
  `pp user add`.
- **NF5** No role-based permissions in v1. Any active registered identity
  MAY call any signed endpoint. A `roles[]` field on the `identity` tuple
  payload is reserved for future use but is not read by v1.
- **NF6** No TLS client-certificate auth. Ed25519 message signing is the
  transport-agnostic equivalent. The underlying HTTP transport MAY be
  TLS-terminated by a reverse proxy, but that is out of scope.
- **NF7** No PP-to-PP federation. One deployment = one PP server. Joining
  two deployments is not supported.
- **NF8** No distributed workflow execution. The Petri-net firing loop in
  `WorkflowEngine` and the tick loop in `Dispatcher>>runLoop` remain
  centralized on the server; only **task EXECUTION** distributes.
- **NF9** No GUI for user administration in v1. `pp user add` / `revoke`
  are CLI-only, operator-driven.
- **NF10** No billing/invoicing. Budget enforcement is local to each worker;
  the server tracks usage for attribution/visibility only.

## 6. Resolved Design Decisions

Resolved via interactive walkthrough on 2026-04-18. Each decision includes
rationale and cross-references the feature IDs it refines. Decisions are
also recorded as `pp decide` tuples in the tuplespace for audit trail
(`pp read decision default` to query).

**D6.1 `witnessed-by` semantics.** Separate signer-supplied field, single
identity (not array). `actor` is always the verified signer — non-negotiable.
`witnessed-by` MAY be set by the signer to any registered identity, meaning
"signer vouches that this identity witnessed the observation."
*Rationale:* preserves audit integrity while supporting "Alice told me X"
reporting. Array support deferred (trivial schema widening later).
*Refines:* F5.1.

**D6.2 Claim lease duration & renewal cadence.** Default lease **10 minutes**,
renewal every **lease/3 (≈3 min)** on a timer (not on progress). Reaper
requires **two consecutive missed renewals past expiry** before reaping.
Not model-dependent; model-awareness deferred. A task MAY declare
`max_runtime` to override the default.
*Rationale:* Opus reviewer tasks commonly exceed 5 min. Err long;
false-reap corrupts worktrees while a slightly-long lease only delays real
crash recovery by minutes. Timer renewal is simpler than progress-detection
and robust to agent behavior.
*Refines:* F3.2, F3.3.

**D6.3 Worker disconnect handling.** Lease expiry is the **single source of
truth** for reaping (option b). Heartbeats feed the dashboard and
early-warning UI but MUST NOT trigger reaping. In-flight worktrees are left
as-is on the disconnected worker; retries create a fresh worktree with a
retry-suffixed branch name: `impl/<task-id>-r<n>`. `retry_count` is
incremented on the task tuple.
*Rationale:* one reaper, one state machine. Two reapers disagree in edge
cases. Retry-suffixed branches prevent late-completion collisions with
retry branches.
*Refines:* F3.3.

**D6.4 Server discovery by workers.** Precedence: **flag > env > config
file**, no mDNS. Config at `~/.config/pp/config.toml` stores both the
server URL and identity path. `pp identity init` SHOULD prompt for
`server` when the config file is absent, bootstrapping both in one step.
*Rationale:* mDNS cost (per-OS daemons, firewall rules, name collisions)
outweighs benefit for a one-line-config problem.
*Refines:* F1.1.

**D6.5 Repo capability verification.** Claim-time filter by **repo name
only** (worker self-declares `repos[]` as eligibility hints). Commit
verification happens at harness start time via `git worktree add <ref>`;
if that fails, emit `task-failed` with reason `repo-not-available` and
release the claim. **`repo-not-available` failures MUST NOT count toward
`max_retries`** — one misconfigured worker must not exhaust retries for
a task.
*Rationale:* execution *is* verification. Server-side challenges are
expensive and stale by runtime.
*Refines:* F2.2, F3.1, F3.3.

**D6.6 Wire protocol.** JSON over HTTP+SSE. Maggie `Node`/`RemoteProcess`
links explicitly deferred. Revisit trigger: measurable >100ms p50 latency
attributable to HTTP overhead, or a need for bidirectional streaming
beyond SSE.
*Rationale:* debuggable with curl, composes with reverse proxies
(Tailscale/Cloudflared/nginx), keeps door open for non-Maggie workers,
avoids coupling to young distributed-runtime features.
*Refines:* all of §4 (transport assumption throughout).

**D6.7 Default dashboard scope.** Split answer:
 - **Presence panel (humans online, workers online):** always shows
   everyone. Low-volume, bounded by team size, it's the point of the
   feature.
 - **Workflow cards:** default to **team** (show all) with a persistent
   per-user `mine | team` toggle.
 - **Notifications:** stay opt-in via `pp watch` (F8.2) — *seeing* others'
   work is cheap, *being interrupted* by it is not.
*Rationale:* collective improvement requires visibility; a "mine only"
default silos users. Noise mitigated by easy toggle + opt-in notifications.
*Refines:* F7.4, F8.1, F8.2.

**D6.8 Worker budget enforcement.** **Hard stop as default.** On exceeding
any configured limit, worker enters soft-drain (stops claiming, finishes
in-flight) and emits `notification` severity `urgent`. **`--budget-mode
soft`** is an opt-in escape hatch: worker continues claiming, emits
severity `warning` on each claim past threshold. In-flight tasks are never
yanked mid-execution (worktree corruption risk). Budget accounting lives
on the worker (authoritative); server records reported usage per task for
attribution only.
*Rationale:* the default must protect the operator's Anthropic balance
from overnight runaways. Soft-warn-only is the wrong default for money.
*Refines:* F2.5.

**D6.9 Key rotation UX.** **Self-rotate via old-key-signed attestation.**
`pp identity rotate` generates a new keypair locally, signs
`{old_pubkey, new_pubkey, timestamp}` with the **old** key, POSTs to
`/api/user/rotate`. Server verifies against registered pubkey, updates
the `identity` tuple, appends an `event` tuple `key-rotated:<actor>` for
audit. Old key stops verifying immediately. Operator `pp user add
--replace` is reserved for **lost-key recovery** (the only case requiring
operator intervention).
*Rationale:* possession of the old private key is the correct authority
for transition. Zero operator involvement for routine rotation, no lockout
window, immediate revocation of old key supports compromise scenarios.
*Refines:* F1.5.

**D6.10 Admin role / bootstrap.** Admin pubkeys listed in
`~/.config/pp/server.toml` under `[admin] admins = ["<hex>", ...]`. Admin
gate applies **only** to `pp user add / revoke / replace`. All other
mutating endpoints are open to any registered active identity (flat per
NF5). On first run, server prints bootstrap instructions if no admins are
configured.
*Rationale:* NF5 governs *between* registered users; *becoming* registered
is a separate trust-root boundary. Config-file admins avoid chicken-and-egg
(can't seed identity tuples via signed writes without first having an
identity).
*Refines:* F1.2, F1.6.

**D6.11 Backpressure.** Unbounded queue by default; dashboard shows queue
depth prominently and renders red when depth exceeds `3 × total_slots`.
Opt-in hard cap via `[queue] max_pending = N` in server config — when hit,
`pp run` returns `429 Too Many Requests` with current depth so the CLI
can surface or retry. No per-launcher fairness in v1 (revisit only if
empirically starving; first lever is talking to the noisy launcher).
*Rationale:* "your work is refused" is a worse default UX than "your work
is queued behind N others, check the dashboard." Claim-side backpressure
(D6.2) already handles the consumer side.
*Refines:* new requirement — add §4.10.

**D6.12 Claim idempotency.** Worker generates a UUIDv4 `claim_id` **before**
sending the claim; retries reuse the same `claim_id`. Server records
`claim_id` on the task tuple when flipping to `claimed`. Retry with
matching `(task_id, claim_id, worker_id)` returns `200 OK` (idempotent
no-op). Different `(worker_id, claim_id)` that already won returns
`409 Conflict` with current `executed-by`. `claim_id` is threaded through
`renew` and `complete` so the server can reject operations from a worker
that thinks it holds a claim it actually lost.
*Rationale:* partitions during claim are the normal case over long enough
timelines. Deterministic IDs absorb them; non-deterministic IDs throw work
away on transient failures.
*Refines:* F2.6, F3.2.

**D6.13 Worker identity keys.** **Distinct keypair per worker**, signed
into existence by an operator-key attestation. `pp worker init` generates
at `~/.config/pp/worker/<name>.key`; `pp worker register` posts worker
pubkey + operator-signed `{worker_pubkey, operator_pubkey, nonce}`
attestation. Server records both on the `worker` tuple. Heartbeat, claim,
renew, complete are signed by worker key. Revocation: **operator-or-admin**
(owners of compromised machines need fast self-serve revocation).
*Rationale:* blast radius. A worker on a public port with a private key
in memory must not be able to sign as its operator. Per-worker scoping
(accept_from, budgets) is the natural policy granularity. Granular
revocation: `pp worker revoke` is cheap and common; human revoke is rare
and heavy.
*Refines:* F2.1, F2.2.

**D6.14 SSE fan-out & backpressure.** Per-connection bounded queue (256
events). When full, server closes the connection with a "reconnect to
resync" signal. Client reconnects, fetches current state via `/api/scan`,
resumes streaming. Instrumented in the existing `ApiServer>>startSSETick`
broadcast loop as a channel-full check.
*Rationale:* SSE lacks flow control; one slow client (sleeping laptop,
background tab) shouldn't block the broadcast loop for everyone.
Drop-and-reconnect is the standard SSE recipe and the client already
fetches fresh state on reconnect.
*Refines:* new requirement — add §4.11.

**D6.15 Workflow cancellation authority.** Launcher, **plus** operators of
any worker currently executing a task in the workflow, plus admins.
`/api/workflow/cancel/<id>` verifies signer is in this set. Cancellation
MUST carry a `--reason` (CLI) / `reason` (API) field; cancelled workflow
emits a `notification` to the launcher with severity `urgent` including
the cancelling actor and reason.
*Rationale:* cancellation is asymmetric — the launcher pays in progress,
executing operators pay in API costs. Bob shouldn't need Alice's consent
to stop her misbehaving scout-mission that's burning his tokens. Open
"any registered user" invites griefing with no benefit.
*Refines:* new requirement — add §4.12.

### Additional requirements from resolved decisions

These refine or extend §4 based on resolutions above.

**4.10 Queue backpressure (D6.11)**
- The dashboard MUST display current pending-task count.
- When `pending_tasks > 3 × sum(worker.slots)`, the dashboard MUST render
  queue depth in a warning color.
- If server config sets `[queue] max_pending = N`, `/api/workflow/run` MUST
  return HTTP 429 with a JSON body `{depth: <current>, max_pending: N}`
  when pending exceeds N.

**4.11 SSE client backpressure (D6.14)**
- Each SSE connection MUST have a bounded per-connection queue (256
  events).
- When the queue is full, the server MUST close the connection with a
  body indicating resync is required.
- The dashboard client MUST, on SSE close, re-fetch current state via
  `/api/scan` before resuming its SSE subscription.

**4.12 Workflow cancellation (D6.15)**
- `POST /api/workflow/cancel/<id>` MUST verify the signer is in
  `{launched-by, any operator whose worker has an active claim on a task
  in this workflow, any admin}`.
- The request body MUST include a `reason` string (min length 1); the CLI
  flag `--reason` is required.
- On successful cancel, the server MUST emit a `notification` tuple to
  the launcher (severity `urgent`) containing the cancelling actor and
  reason.

## 7. Verification Matrix (for future test plan authors)

The following acceptance scenarios test the requirements end-to-end and
should be covered by integration tests under `test/`:

1. **Two users, one server.** Alice and Bob both `pp user add`; both can
   `pp observe`; Bob sees Alice's observations on the dashboard. (G1.1, F1.2)
2. **Unsigned mutation rejected.** A curl `POST /api/out` without headers
   returns 401. (F1.4)
3. **Delegated task.** Alice runs `pp run scout-mission` with
   `--affinity team`; Bob's worker claims and executes; resulting task
   tuple has `launched-by: alice`, `executed-by: worker:<bob's>`.
   (G4.1, F5.1)
4. **Claim race.** Two workers simultaneously claim the same task; exactly
   one succeeds. (F2.6, F3.4)
5. **Worker disconnect.** Kill a worker mid-task; task reappears as
   `pending` after lease expiry and is re-claimable. (F3.3)
6. **Budget soft-stop.** Configure `--max-tokens-per-day 1000`; after
   exceeding, worker stops claiming but completes in-flight work. (F2.5)
7. **Revocation.** `pp user revoke bob`; Bob's next signed request returns
   401. (F1.6)
8. **Replay defense.** Capture a valid signed request, resend after the
   skew window; server returns 401. (F1.4)
9. **Convention visibility.** Alice promotes an observation to a
   convention; Bob's next `pp run` sees it in agent context. (G1.2, G1.4)
10. **Attribution audit.** `pp audit --actor alice` lists only Alice-signed
    tuples; `pp audit --worker worker:<bob>` lists tasks executed by Bob.
    (F6.1, F6.2)
